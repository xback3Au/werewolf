Phase 1（当前优先）：数据结构化 + RAG 底座
========================================
【目标】：把现有 JSON 输出转成关系型表，为 Agent 的工具和记忆打基础

1.1 建立新数据库表（在现有 analyses/uploads 基础上扩展）

  - speeches 表（发言详情）
    analysis_id, day, speaker_id, text_summary, full_text,
    stance_json, intent_tags, embedding(BLOB), impact_score, is_key_moment

  - votes 表（票型）
    analysis_id, day, voter_id, target_id, vote_reason

  - timeline_events 表（时间线锚点）
    analysis_id, day, phase, event_type, content, is_public, visible_to

  - inference_chains 表（推理链，是 Agent 学习的核心）
    analysis_id, observer_id, observer_role, target_id,
    conclusion, confidence, reasoning, evidence_ids,
    is_correct(用户反馈后填), truth, error_type

  - knowledge_chunks 表（经验库，跨局积累）
    chunk_type(strategy/mistake/insight), content, embedding,
    game_context, importance_score, is_teaching_material

1.2 双写方案（不破坏现有流程）
  Stage2/3 分析 → 同时写 JSON（给前端）+ INSERT 到新表（给 Agent 查询）

1.3 实现 GameState 类（局内状态追踪）
  - alive_players, dead_players（死亡方式）
  - confirmed_roles（已确认身份）
  - claimed_roles（声称身份记录）
  - vote_history
  - stance_graph（立场关系图：谁攻击谁、谁保谁）

【完成标志】：能从数据库按"第几天/第几号玩家"精确查出发言+投票+死亡信息

========================================
Phase 2：核心工具（Tool）开发
========================================
【原则】：先写工具 → 每个独立测试通过 → 再串进 Agent

Tool 1: game_parser_tool（对局解析）
  输入：原始聊天记录文本
  输出：结构化 JSON（玩家列表、轮次、发言、投票、死亡）
  测试：给一局样本文本，验证输出格式稳定

Tool 2: logic_checker_tool（逻辑矛盾检测）
  输入：某玩家所有发言
  输出：矛盾点列表（"前一轮保3号，后一轮投3号"）
  测试：构造有矛盾的发言，验证能否检出

Tool 3: identity_inference_tool（身份概率推理，核心）
  输入：某玩家行为 + 发言
  输出：{狼面%: xx, 好人面%: xx, 神面%: xx, 依据: [...]}
  测试：用已知身份的对局验证准确率

Tool 4: tactic_summarizer_tool（战术归纳）
  输入：完整分析结果（含真实身份）
  输出：本局好人打法要点、狼人胜利关键操作、可复用技巧

Tool 5: feedback_parser_tool（解析用户纠正）
  输入：用户自然语言（"5号其实是狼，他在倒钩"）
  输出：{target_player: 5, real_role: "wolf", behavior: "倒钩", reason: "..."}
  测试：各种口语化纠正表述都能正确解析

Tool 6: memory_update_tool（经验库更新）
  输入：结构化纠正信息
  动作：更新 inference_chains.is_correct 和 truth
        生成一条经验规则存入 knowledge_chunks
  输出：生成的经验描述（"发言阳光但投票异常 → 大概率倒钩狼"）

【完成标志】：6 个 Tool 各自有测试用例，独立调用均返回预期格式

========================================
Phase 3：Agent 主体（ReAct 循环）
========================================
【时机】：Phase 2 所有 Tool 测试通过后再开始

3.1 Agent 规划流程（固定骨架）
  1. 接收输入（对局文本 或 用户聊天消息）
  2. 调用 game_parser_tool → 结构化数据
  3. 多轮推理循环：
     - 调用 logic_checker_tool 逐人分析
     - 调用 identity_inference_tool 标身份概率
  4. 调用 tactic_summarizer_tool → 战术总结
  5. 输出分析报告

3.2 用户纠正分支（会话中途触发）
  用户说"你判断错了" → Agent 进入修正规划：
  → 调用 feedback_parser_tool 解析纠正
  → 更新本局短期记忆（inference_chains）
  → 调用 memory_update_tool 生成经验存入长期记忆
  → 重新推导受影响的判断 → 输出修正后结论

3.3 技术选型
  框架：LangChain / 自己写 ReAct 循环（推荐后者，更好控制）
  规划模式：ReAct（思考→行动→观察→更新）
  Tool 注册：Python function + docstring 描述

【完成标志】：丢一局完整对局，Agent 能自主走完全流程输出学习报告

========================================
Phase 4：长期记忆 + 越学越强
========================================
4.1 经验库检索（RAG）
  - 分析新对局前，先从 knowledge_chunks 检索相关经验
  - 检索方式：向量语义搜索（Chroma 或 SQLite-vss）
  - 效果：遇到类似打法，自动调用已有经验，不再犯同样错误

4.2 跨局统计
  - 某种发言模式的历史准确率
  - 哪类打法对好人最有效 / 最危险
  - 生成"个人风格画像"（如果有固定玩家）

4.3 前端对话界面（可选）
  - 在现有界面新增"对话复盘"标签页
  - 用户可与 Agent 多轮对话，实时纠正、提问

========================================
关键技术决策备忘
========================================
1. Tool 和 Agent 分离：Tool 负责具体逻辑，Agent 只负责调度
2. 短期记忆：单局对话上下文（JSON/内存）
   长期记忆：knowledge_chunks 表（持久化）
3. 向量数据库选型：优先 Chroma（本地，轻量），需要生产环境再考虑 Pinecone
4. 不破坏现有流水线：Phase 1-4 全部叠加在现有架构上，老功能继续工作
